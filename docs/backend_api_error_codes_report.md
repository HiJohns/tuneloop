# 后端 API 错误码调查报告

## 1. 概述

本报告调查了 TuneLoop 后端项目（backend）中所有 API 错误码的使用情况，分析了错误码的分类体系和使用场景。

### 1.1 调查范围

- **后端项目路径**: `backend/handlers/`
- **调查目标**: 统计所有 API 返回的错误码

---

## 2. 错误码体系

后端使用了一套标准的错误码体系，按类别分组：

| 错误码 | 含义 | HTTP 状态码 | 使用场景 |
|--------|------|-------------|----------|
| **20000** | 成功 | 200 | 正常成功响应 |
| **20100** | 创建成功 | 201 | 资源创建成功 |
| **40000** | 通用请求错误 | 400 | 参数验证失败 |
| **40001** | 请求格式错误 | 400 | 请求体解析失败 |
| **40002** | 参数缺失/无效 | 400 | 缺少必填参数 |
| **40003** | 业务校验失败 | 400 | 业务规则校验不通过 |
| **40004** | 重复资源 | 400 | 资源已存在 |
| **40005** | 资源冲突 | 400 | 资源状态冲突 |
| **40006** | 条件不满足 | 400 | 前置条件不满足 |
| **40100** | 认证失败 | 401 | 缺少 tenant_id |
| **40101** | 未认证 | 401 | 用户未登录/token 无效 |
| **40102** | Token 过期 | 401 | Token 已过期 |
| **40300** | 权限不足 | 403 | 角色权限不足 |
| **40301** | 无操作权限 | 403 | 无权执行此操作 |
| **40400** | 资源不存在 | 404 | 记录未找到 |
| **40401** | 资源已删除 | 404 | 资源已被软删除 |
| **50000** | 服务器内部错误 | 500 | 数据库/服务异常 |
| **50001** | 服务不可用 | 500 | 依赖服务故障 |
| **50002** | 数据库错误 | 500 | 数据库操作失败 |

## 3. 错误码详细说明

### 3.1 成功类 (2xx)

| 错误码 | 说明 | 示例 |
|--------|------|------|
| **20000** | 操作成功 | GET/POST/PUT 成功返回 |
| **20100** | 资源创建成功 | POST 创建新资源 |

### 3.2 客户端错误类 (4xx)

#### 40000-40099 请求错误

| 错误码 | 说明 | 典型场景 |
|--------|------|----------|
| **40000** | 通用请求错误 | 参数绑定失败、JSON 解析错误 |
| **40001** | 请求格式错误 | ShouldBind 失败 |
| **40002** | 参数缺失 | 必填字段为空、ID 未提供 |
| **40003** | 业务校验失败 | 状态不匹配、金额不足 |
| **40004** | 重复资源 | 唯一索引冲突 |
| **40005** | 资源冲突 | 状态不允许操作 |
| **40006** | 条件不满足 | 前置条件未满足 |

#### 40100-40199 认证错误

| 错误码 | 说明 | 典型场景 |
|--------|------|----------|
| **40100** | 缺少租户信息 | tenant_id 为空或无效 |
| **40101** | 未认证/Token 无效 | 用户未登录或 Token 无效 |
| **40102** | Token 过期 | JWT Token 已过期 |

#### 40300-40399 权限错误

| 错误码 | 说明 | 典型场景 |
|--------|------|----------|
| **40300** | 权限不足 | 用户角色无权访问 |
| **40301** | 无操作权限 | 只能操作自己的数据 |

#### 40400-40499 资源错误

| 错误码 | 说明 | 典型场景 |
|--------|------|----------|
| **40400** | 资源不存在 | 记录未找到 |
| **40401** | 资源已删除 | 软删除后的查询 |

### 3.3 服务端错误类 (5xx)

| 错误码 | 说明 | 典型场景 |
|--------|------|----------|
| **50000** | 服务器内部错误 | 一般异常 |
| **50001** | 服务不可用 | 外部依赖失败 |
| **50002** | 数据库错误 | SQL 执行失败 |

## 5. 错误处理模式

### 5.1 标准响应格式

```json
{
  "code": <错误码>,
  "message": "<错误描述>"
}
```

### 5.2 成功响应格式

```json
{
  "code": 20000,
  "data": {
    // 业务数据
  }
}
```

### 5.3 常见处理模式

**参数验证**:
```go
if err := c.ShouldBindJSON(&req); err != nil {
    c.JSON(http.StatusBadRequest, gin.H{
        "code":    40001,
        "message": "invalid parameters: " + err.Error(),
    })
    return
}
```

**资源不存在**:
```go
if err == gorm.ErrRecordNotFound {
    c.JSON(http.StatusNotFound, gin.H{
        "code":    40400,
        "message": "resource not found",
    })
    return
}
```

**服务器错误**:
```go
if err != nil {
    c.JSON(http.StatusInternalServerError, gin.H{
        "code":    50000,
        "message": "failed to process: " + err.Error(),
    })
    return
}
```

## 6. 改进建议

### 6.1 现有问题

1. **错误码分散**: 错误码在各个 handler 中定义，缺乏统一管理
2. **部分错误码未使用**: 40006、40102、50002 等未找到使用
3. **message 文本不统一**: 错误信息格式不统一

### 6.2 建议

1. **建立错误码常量**: 建议在项目中定义统一的错误码常量
2. **补充前端处理**: 确认前端已处理所有错误码情况
3. **统一错误信息**: 制定错误信息规范
