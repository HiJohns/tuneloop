# TuneLoop UI 设计文档

## 1. 概述

### 1.1 文档目的
本文档定义 TuneLoop 乐器租赁管理系统的界面设计规范，包括页面结构、功能点、交互逻辑和视觉设计原则。

### 1.2 技术栈
- **前端框架**: React 18 + Vite
- **UI 组件库**: Ant Design 5.x
- **路由**: React Router 6
- **状态管理**: React Hooks (useState, useContext)
- **样式**: Tailwind CSS + Ant Design 样式覆盖

### 1.3 页面入口
- **PC 管理后台**: http://localhost:5554
- **移动端**: http://localhost:5556

## 3. 页面设计

### 3.0 冷启动向导 (Setup)

**路由**: `/setup`  
**权限**: 无需登录（仅限未初始化系统访问）

**页面流程**:

1. **状态检测**: 页面加载时调用 `GET /api/setup/status`
   - 若 `requires_setup = false` → 自动重定向至登录页 `/`
   - 若 `requires_setup = true` → 显示初始化表单

2. **初始化表单**:
   - 邮箱（输入框，带格式验证）
   - 密码（密码框，强度提示）
   - 确认密码（密码框，一致性验证）
   - 『创建管理员』按钮（提交）

3. **提交处理**:
   - 表单验证通过后调用 `POST /api/setup/init`
   - 显示加载状态
   - 成功后后端返回 OIDC 授权 URL
   - 前端自动跳转至 IAM 完成首次认证

4. **错误处理**:
   - 系统已初始化（403）→ 显示错误并跳转登录页
   - 参数错误（400）→ 高亮显示错误字段

**交互细节**:
- 表单验证实时反馈
- 密码强度可视化（弱/中/强）
- 提交按钮禁用状态管理

### 3.1 商户管理 (MerchantManagement)

**路由**:
- 列表: `/merchants`
- 创建: `/merchants/new`
- 详情: `/merchants/:id`

**权限**: 仅 `project_admin` 角色可见

#### 3.1.1 商户列表页

**页面结构**:
- 顶部工具栏
  - 搜索框（按商户名称筛选）
  - 『创建商户』按钮（右上角）
- 表格区域
  - 列：商户名称、商户代码、联系人、管理员、创建时间、状态、操作
  - 行操作：查看详情、编辑、删除

**交互**:
- 分页加载（每页 20 条）
- 点击行进入详情页
- 删除操作弹出确认对话框，包含资产校验结果提示

#### 3.1.2 商户创建页

**表单字段**:
- 商户名称（必填）
- 商户代码（必填，格式验证，唯一性校验）
- 联系人信息：
  - 姓名
  - 邮箱（格式验证）
  - 电话（手机号格式验证）
- 指定管理员：
  - 点击『选择管理员』按钮
  - 弹出「指定用户对话框」（见 §3.2）
  - 选择后显示用户姓名和邮箱

**验证规则**:
- 商户代码：仅允许字母、数字、连字符
- 邮箱：标准邮箱格式
- 电话：11 位手机号

**提交**:
- 校验通过后调用 `POST /api/merchants`
- 成功后跳转至商户列表页

**错误处理**:
- 代码重复 → 提示"商户代码已存在"
- 管理员用户不存在 → 提示"请重新选择管理员"

### 3.2 指定用户对话框 (UserSelectionDialog)

**组件类型**: 可复用模态对话框

**使用场景**:
- 商户创建：选择管理员
- 网点管理：添加成员
- 人员管理：指定归属

**对话框结构**:

1. **搜索区域**:
   - 输入框（提示：输入用户名、姓名、邮箱或手机）
   - 『搜索』按钮

2. **结果展示区**（根据搜索结果显示不同状态）:

   **状态 A：用户已存在且属于当前商户**
   ```
   ┌─────────────────────────────┐
   │ ✓ 找到匹配用户              │
   │ 姓名：张三                  │
   │ 邮箱：zhangsan@example.com  │
   │                             │
   │ [确认选择]                  │
   └─────────────────────────────┘
   ```

   **状态 B：用户存在但不属于当前商户**
   ```
   ┌──────────────────────────────────────┐
   │ ⚠ 该用户已在 Tuneloop 注册           │
   │ 姓名：李四                           │
   │ 邮箱：lisi@example.com               │
   │                                      │
   │ 是否邀请其加入本商户？               │
   │ [取消] [邀请并选择]                  │
   └──────────────────────────────────────┘
   ```

   **状态 C：用户完全不存在**
   ```
   ┌──────────────────────────────────────┐
   │ ✗ 未找到用户                         │
   │                                      │
   │ 是否立即创建新用户？                 │
   │                                      │
   │ 姓名：[________]                     │
   │ 邮箱：[________]                     │
   │ 手机：[________]                     │
   │ 初始密码：[________]                 │
   │                                      │
   │ [取消] [创建并选择]                  │
   └──────────────────────────────────────┘
   ```

3. **按钮组**:
   - 取消：关闭对话框
   - 确认/邀请/创建：执行相应操作并返回 user_id + user_name

**交互细节**:
- 支持 Enter 键快速搜索
- 表单实时验证
- 创建用户时邮箱/手机号唯一性校验

- 描述信息

  - 上传进度显示
  - 失败重试
- 视频 URL

**表单验证**:
- 识别码: 必填，唯一性校验

**规则保护**:
- 最后一名管理员：
  - 切换角色按钮禁用（灰色）
  - 移除按钮禁用（灰色）
  - Hover 显示 Tooltip: "最后一名管理员不可修改"

#### 3.10.2 删除网点增强

**删除前校验**:
- **资产检查**: 调用库存接口检查
  - 在库乐器（available）→ 弹窗提示: "该网点仍有 X 件在库乐器，请先转移资产"
  - 在租乐器（rented）→ 弹窗提示: "该网点仍有 X 件在租乐器，请先处理订单"
- **成员检查**: 若仍有成员 → 提示: "请先移除所有成员"

**删除流程**:
- 全部检查通过后弹出确认对话框
- 确认后调用 `DELETE /api/merchant/sites/:id`
- 成功后：左侧树移除节点，路由跳转至 `/sites`

### 3.16 维修会话 (MaintenanceSession)

**路由**: `/maintenance/sessions`

**权限**: 各角色不同视图
- 技师: 查看自己所属订单
- 员工: 查看网点所有订单

**功能点**:
- 维修订单列表
  - 日期、类别、简述、状态
  - 待分配状态显示『接单』按钮
  - 已分配状态显示在手订单标识
- 维修详情页
  - 问题描述和图片
  - 进度备注
  - 维修报告
  - 维修照片
- 技师操作（微信端）
  - 点击『接单』（待分配→已分配）
  - 点击『开始工作』（维修中状态）
  - 扫码乐器二维码
  - 输入评论、上传照片
  - 点击『完成』（验收状态）
- 员工验收（微信端）
  - 扫码乐器二维码
  - 按规定拍照上传
  - 点击『验收通过』（订单结束，在库状态）
  - 点击『验收不通过』（添加评论，待处理状态）

**状态流转**:
- 待分配 → 已分配（技师接单）
- 已分配 → 维修中（师傅开始工作）
- 维修中 → 验收中（师傅完工）
- 验收中 → 在库（员工验收通过）
- 验收中 → 待处理（员工验收不通过）

**交互逻辑**:
- 状态变更需记录操作人和时间
- 照片上传使用移动端相机组件
- 状态不可逆（除验收不通过回退到待处理）
- 通知机制：状态变更推送消息给相关人员

### 3.17 申诉处理 (AppealManagement)

**路由**: `/appeals`
